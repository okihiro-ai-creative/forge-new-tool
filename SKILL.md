# Forge New Tool (Agent Edition)

このスキルは、Claude Code / Antigravity等の**AIエージェントがファイルシステムを直接操作できること**を前提に、実務で使えるPythonツールを構築するための手順書です。

チャットUIで生成したコードを人間がコピペで反映する運用とは異なり、エージェントが自分でファイルを読み書き・実行できるため、コード生成→ファイル反映という中間工程は不要です。エージェントが直接ファイル一式を作成します。

## 1. 要件定義（Phase 0: Deep-Dive Dialogue）

**最重要:** ユーザーがツールの要望を伝えてきた際、即座に構築を開始してはならない。

1. **深掘り対話**: 以下の観点を中心にユーザーと対話し、理想の形を明確にする。
   - **ターゲット**: 誰が、いつ、どこで使用するツールか。
   - **コア体験**: そのツールを使うことで解決したい最大の課題は何か。
   - **機能詳細**: 必要な入力、期待する出力、GUIに求める操作感。
2. **設計案の提示**: 対話に基づき、どのようなツールを作るか（構成・機能一覧）を要約して提示し、ユーザーの承認を得ること。

## 2. 格納ルールの厳守

- **ルート基準**: このスキル実行時のカレントディレクトリを基準とする。
- 新規ツールの作成場所は、プロジェクトルート直下のツール格納フォルダに限定する。
  - 推奨例: `./tools/{ToolName}` または `./apps/{ToolName}`
- **重要**: 既存の設定フォルダ・スキル本体フォルダ・認証情報フォルダへのツール作成は禁止。

## 3. フレームワーク 必須要件

エージェントは以下の要件を満たすファイル一式を、ツール格納フォルダ直下に直接生成する。

1. **モデル選定と管理**: ツールがGemini APIを呼び出す場合、`system/config.ini` またはGUIのSettingsタブで使用モデルを選択できるプルダウンを実装する。
   - **重要**: モデル名は本SKILL.mdに固定で書かれた値を鵜呑みにせず、実行の都度 [Gemini API 料金ページ](https://ai.google.dev/gemini-api/docs/pricing)（公式・一次情報）で現行モデルを確認すること。廃止済みモデルは使用禁止。
2. **プロンプト外部化**: AIへの指示文（プロンプト）はコード内にハードコードせず、`prompts/` フォルダ内にテキストファイル（例: `prompt_template.txt`）として作成し、プログラムから読み込む設計にする。
3. **構成**: `gui/modules`, `scripts/runners`, `input`, `output`, `system`, `lib`, `prompts` に分割する。
4. **ライブラリ**: 旧来の `google-generativeai` ではなく、最新の `google-genai` SDKを使用する。
5. **APIキー管理（Gemini APIを使う場合）**: `lib/api_manager.py` にAPIキー自動ローテーション機能を実装する。
   - `system/api_key.txt` から改行区切りで複数のAPIキーをリストとして読み込む。
   - `system/.api_key_state.json` に前回使用したキーのインデックスを記録・永続化する。
   - 起動時に、保存されたインデックスの次のキーを選択する（ラウンドロビン）。
   - APIエラー（429等）発生時、自動的に次のキーに切り替えてリトライする。
6. **GUI**: `ttkbootstrap` を使用し、重い処理は `threading` で非同期実行する。起動時にウィンドウが画面中央に表示されるようにする。
7. **デバッグ環境（必須）**: `_run_debug.bat` を作成する。内容には `chcp 65001`、`.venv` の存在確認、`python -u`（バッファ無効化）、終了後の `pause` を含める。
8. **セットアップの分離**: `_setup.bat`（仮想環境作成）と `_run.bat`（起動）は必ず別ファイルとして作成する。統合禁止。
9. **f-string使用禁止（NameError対策・最重要）**: あるPythonファイルが別のPythonファイルの中身を文字列として生成する場合、f-string（`f"""..."""`）は使用しない。
   - **理由**: 出力対象のコード内に `{}` が含まれていた場合、実行時に変数が評価されて `NameError` が発生するため。
   - **対策**: 通常の文字列（`"""..."""`）を使用する。変数を埋め込む必要がある場合は、波括弧を二重（`{{ }}`）にしてエスケープするか `.format()` を使う。

## 4. 鍛造プロセス（エージェント直接実装版）

### Step 1: ファイル一式の生成
上記フレームワーク要件を満たすファイル一式を、エージェントがツール格納フォルダに直接作成する。
- ターゲットディレクトリが存在しない場合は作成すること。

### Step 2: 環境構築
`_setup.bat` を実行する。

    call _setup.bat

### Step 3: 起動確認（Ignition）
デバッグ用バッチを実行して、エラーなくGUIが起動することを確認する。
(GUIが起動確認できたタイミング、あるいはログに `Listening` 等が表示されたら成功とみなす)

    call _run_debug.bat

### Step 4: 機能追加（継続開発）
一度動くものができた後は、全体を再生成しない。ファイル単位の個別修正で機能を追加していく。
「このファイルのこの機能を追加/修正して」という粒度でエージェントに依頼する。

## 5. エラー対応

エラーが出た場合は、エラーメッセージをそのままエージェントに渡し、該当ファイルの修正コード全文を提示してもらう。

## 6. 完了報告

- 作成されたツールのパス
- 起動確認の結果
これらを報告して終了する。
